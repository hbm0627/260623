# 개인 홈페이지 서비스 기획서

## 1. 서비스 개요

- 서비스명: 개인 홈페이지 서비스
- 한 줄 소개: 개인 소개, 게시판, 회원 기능, 관리자 기능을 포함한 개인 웹사이트
- 기획 배경: 개인 포트폴리오와 커뮤니티형 게시판을 함께 운영할 수 있는 웹 서비스를 구축한다.
- 핵심 가치:
  - 방문자는 메인 페이지에서 운영자 정보를 빠르게 확인할 수 있다.
  - 회원은 가입, 로그인 후 게시판을 이용할 수 있다.
  - 사용자는 자신의 프로필 정보를 관리할 수 있다.
- 관리자는 어드민 계정으로만 관리자 페이지에 진입해 게시물, 댓글, 대댓글을 관리할 수 있다.

## 2. 목표 사용자

- 주요 사용자:
  - 홈페이지 방문자
  - 회원가입 후 게시판을 이용하는 일반 사용자
  - 사이트를 운영하고 관리하는 관리자
- 사용자 문제:
  - 운영자의 소개, 활동, 게시글을 한 사이트에서 확인하기 어렵다.
  - 게시판 운영 시 사용자 및 게시물 관리 기능이 필요하다.
  - 로그인 기반 기능과 관리자 권한 구분이 필요하다.
- 사용 상황:
  - 방문자가 홈페이지에 접속해 소개 정보를 확인한다.
  - 사용자가 회원가입 후 로그인한다.
  - 로그인한 사용자가 게시글을 작성, 조회, 수정, 삭제한다.
  - 관리자가 어드민 계정으로 로그인한 뒤 관리자 페이지에서 게시물, 댓글, 대댓글을 관리한다.

## 4. 페이지 구성

| 페이지 | 경로 예시 | 접근 권한 | 설명 |
| --- | --- | --- | --- |
| 메인 페이지 | `/` | 전체 | 개인 소개, 주요 링크, 최신 게시글 일부를 보여준다. |
| 회원가입 페이지 | `/signup` | 비로그인 | 이름, 이메일, 비밀번호를 입력해 회원가입한다. |
| 로그인 페이지 | `/login` | 비로그인 | 이메일과 비밀번호로 로그인한다. |
| 프로필 페이지 | `/profile` | 로그인 사용자 | 내 정보 조회 및 수정, 내가 작성한 게시글 확인 기능을 제공한다. |
| 게시판 목록 페이지 | `/boards` | 전체 또는 로그인 사용자 | 게시글 목록, 검색, 페이지네이션을 제공한다. |
| 게시글 상세 페이지 | `/boards/:id` | 전체 또는 로그인 사용자 | 게시글 상세 내용을 확인한다. |

| 게시글 수정 페이지 | `/boards/:id/edit` | 작성자 본인 | 기존 게시글을 수정한다. |
| 관리자 로그인 페이지 | `/admin/login` | 비로그인 | 관리자 계정으로 로그인한다. 일반 유저 계정은 접근할 수 없다. |
| 관리자 페이지 | `/admin` | 관리자 | 유저가 생성한 게시물, 댓글, 대댓글을 관리한다. |

## 6. 게시판 기능 상세

### 게시글 기능

- 게시글은 생성, 조회, 수정, 삭제가 가능해야 한다.
- 게시글 목록은 비회원도 조회할 수 있다.
- 게시글 상세 페이지는 비회원도 조회할 수 있다.
- 게시글 작성은 로그인한 유저만 가능하다.
- 게시글 수정은 해당 게시글을 생성한 유저만 가능하다.
- 게시글 삭제는 해당 게시글을 생성한 유저만 가능하다.
- 관리자는 관리자 페이지에서 운영 목적상 게시글을 관리할 수 있다.
- 게시글 상세 페이지에 접근하면 조회수가 증가한다.
- 조회수는 게시글 목록과 상세 페이지에서 표시할 수 있다.

### 댓글 및 대댓글 기능

- 게시글 상세 페이지에서 댓글을 작성할 수 있어야 한다.
- 댓글 작성은 로그인한 유저만 가능하다.
- 댓글에는 대댓글, 즉 답글을 작성할 수 있어야 한다.
- 대댓글 작성은 로그인한 유저만 가능하다.
- 댓글과 대댓글은 작성자 정보, 내용, 작성일, 수정일을 포함한다.
- 댓글과 대댓글의 수정 및 삭제는 작성자 본인만 가능하도록 한다.
- 관리자는 관리자 페이지 또는 관리 API를 통해 댓글과 대댓글을 관리할 수 있다.

### 좋아요 및 싫어요 기능

- 댓글과 대댓글에는 좋아요 기능이 있어야 한다.
- 댓글과 대댓글에는 싫어요 기능이 있어야 한다.
- 좋아요와 싫어요는 로그인한 유저만 누를 수 있다.
- 한 유저는 하나의 댓글 또는 대댓글에 좋아요와 싫어요 중 하나만 선택할 수 있다.
- 이미 누른 반응을 다시 누르면 취소할 수 있다.
- 좋아요를 누른 상태에서 싫어요를 누르면 기존 좋아요는 취소되고 싫어요로 변경된다.
- 싫어요를 누른 상태에서 좋아요를 누르면 기존 싫어요는 취소되고 좋아요로 변경된다.

## 7. 관리자 기능 상세

### 관리자 접근 정책

- 관리자 페이지는 어드민 계정으로 로그인한 사용자만 접근할 수 있어야 한다.
- 일반 회원 계정으로는 관리자 로그인에 성공할 수 없거나, 로그인 후 관리자 페이지 접근이 차단되어야 한다.
- 프론트엔드는 `AdminRoute`에서 로그인 여부와 사용자 권한이 `admin`인지 확인한다.
- 백엔드는 모든 관리자 API에서 JWT 검증 후 `role` 값이 `admin`인지 확인한다.
- 관리자 권한이 없는 사용자가 `/admin` 또는 `/api/admin/*`에 접근하면 접근 거부 응답을 반환한다.

### 관리자 게시물 관리

- 관리자는 유저가 생성한 전체 게시물 목록을 조회할 수 있다.
- 게시물 목록에는 제목, 작성자, 작성일, 수정일, 조회수, 삭제 상태가 표시되어야 한다.
- 관리자는 특정 게시물의 상세 내용을 확인할 수 있다.
- 관리자는 운영 정책에 따라 게시물을 수정하거나 삭제할 수 있다.
- 게시물 삭제는 실제 삭제 또는 `isDeleted` 값을 이용한 소프트 삭제 방식 중 하나로 구현한다.
- 관리자 게시물 관리는 일반 사용자의 “본인 게시글만 수정/삭제” 규칙과 분리된 운영자 권한이다.

### 관리자 댓글 및 대댓글 관리

- 관리자는 유저가 생성한 전체 댓글과 대댓글 목록을 조회할 수 있다.
- 댓글 목록에는 원본 게시글, 작성자, 내용, 좋아요 수, 싫어요 수, 작성일, 삭제 상태가 표시되어야 한다.
- 관리자는 특정 게시글에 달린 댓글과 대댓글을 함께 확인할 수 있다.
- 관리자는 부적절한 댓글 또는 대댓글을 삭제할 수 있다.
- 댓글과 대댓글 삭제는 실제 삭제 또는 `isDeleted` 값을 이용한 소프트 삭제 방식 중 하나로 구현한다.
- 댓글과 대댓글의 좋아요/싫어요 기록은 관리 목적 조회 대상에 포함할 수 있다.

## 8. 디자인 및 사용자 경험 요구사항

### 디자인 방향

- 홈페이지는 개성이 드러나는 나만의 웹페이지로 디자인한다.
- 단순한 템플릿형 개인 페이지가 아니라, 운영자의 성향과 분위기가 느껴지는 시각 스타일을 만든다.
- 메인 페이지 첫 화면에서 로고, 프로필 이미지, 배경 이미지, 대표 문구가 명확하게 보이도록 구성한다.
- 게시판 기능이 포함되어 있으므로 과도한 장식보다 읽기 쉬운 레이아웃과 개성 있는 비주얼의 균형을 맞춘다.
- 전체 톤은 레퍼런스 조사 후 하나의 디자인 콘셉트로 정리한다.

### UX/UI 레퍼런스 조사

- 다른 포트폴리오, 개인 홈페이지, 커뮤니티형 웹사이트의 UX/UI를 참고한다.
- 참고 시 그대로 복사하지 않고 색상, 여백, 타이포그래피, 섹션 흐름, 인터랙션 방식만 분석해 내 프로젝트에 맞게 재구성한다.
- 레퍼런스 웹페이지는 스크린샷으로 저장하고, 각 스크린샷에서 다음 항목을 기록한다.
  - 주요 컬러
  - 배경 분위기
  - 폰트 크기와 제목 스타일
  - 버튼과 링크 스타일
  - 모바일 메뉴 방식
  - 스크롤 또는 전환 인터랙션
- 좋은 점은 내 홈페이지에 반영하고, 불편한 점은 보완해 더 쓰기 쉬운 UI로 만든다.
- 레퍼런스에서 추출한 색상은 그대로 고정하지 않고, 내 프로필 이미지와 배경 이미지에 어울리도록 조정한다.

### 스크린샷 활용 방식

- 참고 웹페이지의 데스크톱 화면과 모바일 화면을 각각 스크린샷으로 남긴다.
- 스크린샷은 디자인 분석용으로만 사용하고, 실제 서비스에는 원본 이미지를 그대로 사용하지 않는다.
- 스크린샷을 보며 색상 팔레트, 섹션 구성, 버튼 상태, 카드 스타일, 메뉴 구조를 정리한다.
- 구현 후 내 홈페이지도 데스크톱과 모바일 화면을 스크린샷으로 확인한다.
- 스크린샷 점검 시 텍스트 겹침, 버튼 크기, 이미지 잘림, 모바일 메뉴 동작, 댓글/대댓글 영역 가독성을 확인한다.

### 이미지 및 로고 자산

- 홈페이지에 필요한 사진, 이미지, 로고는 Codex 내장 이미지 생성 스킬을 활용해 제작한다.
- 생성 대상 예시는 다음과 같다.
  - 메인 페이지 히어로 이미지
  - 개인 홈페이지 로고
  - 프로필 또는 소개 섹션용 이미지
  - 배경 이미지
  - 게시판 빈 상태 이미지
  - 관리자 페이지용 보조 이미지 또는 아이콘 이미지
- 생성된 이미지는 프론트엔드 프로젝트의 정적 자산 폴더에 저장한다.
- 권장 저장 위치는 `frontend/src/assets` 또는 `frontend/public/images`이다.
- 실제 서비스에서 사용하는 이미지는 외부 임시 경로가 아니라 프로젝트 내부 경로를 참조해야 한다.
- 로고는 작은 화면에서도 식별 가능해야 하며, 헤더와 모바일 메뉴에 모두 사용할 수 있어야 한다.
- 이미지에는 불필요한 워터마크, 읽기 어려운 텍스트, 저작권 문제가 있는 브랜드 요소를 넣지 않는다.
- 프로필 이미지는 운영자 개인의 분위기를 표현하되, 실제 사진 또는 AI 생성 스타일 중 하나를 선택한다.
- 배경 이미지는 콘텐츠 가독성을 해치지 않도록 어둡기, 대비, 여백을 조정한다.
- 이미지 생성 전에는 레퍼런스 스크린샷에서 추출한 컬러와 분위기를 프롬프트에 반영한다.

### 모바일 반응형

- 홈페이지는 모바일 페이지를 반드시 지원한다.
- 모바일, 태블릿, 데스크톱 기준으로 레이아웃이 자연스럽게 전환되어야 한다.
- 모바일에서는 헤더 메뉴를 햄버거 메뉴 또는 하단 내비게이션 형태로 제공할 수 있다.
- 게시판 목록, 게시글 상세, 댓글, 대댓글은 모바일에서도 읽기 쉽고 조작하기 쉬워야 한다.
- 관리자 페이지는 모바일에서도 접근 가능하되, 표 형태 데이터는 카드형 목록 또는 가로 스크롤을 적용한다.
- 입력 폼은 모바일 화면에서 버튼과 입력창이 겹치지 않도록 구성한다.
- 주요 터치 영역은 충분한 크기를 확보한다.

### 인터랙티브 UI

- 버튼, 링크, 입력 폼, 메뉴에는 hover, focus, active 상태를 제공한다.
- 게시글 작성, 수정, 삭제 시 로딩 상태와 성공/실패 피드백을 제공한다.
- 댓글과 대댓글 작성 후에는 목록이 즉시 갱신되거나 갱신 상태를 명확히 보여준다.
- 좋아요와 싫어요 버튼은 선택 상태가 시각적으로 구분되어야 한다.
- 게시글 삭제, 댓글 삭제, 관리자 삭제 기능에는 확인 모달을 적용한다.
- 관리자 페이지는 게시물, 댓글, 대댓글 관리 탭을 제공한다.
- 페이지 전환은 과하지 않은 애니메이션으로 자연스럽게 처리할 수 있다.
- 모바일 메뉴 열기/닫기, 댓글 펼침/접힘, 대댓글 입력창 표시 같은 상호작용을 제공한다.
- 메인 페이지에는 스크롤 반응형 요소, 섹션 진입 애니메이션, 이미지 hover 효과 등 가벼운 인터랙션을 적용한다.
- 접근성을 위해 인터랙션은 마우스 hover에만 의존하지 않고 키보드 focus와 터치 환경에서도 동작해야 한다.

## 10. 사용자 흐름

### 일반 방문자 흐름

1. 사용자가 메인 페이지에 접속한다.
2. 개인 소개와 최신 게시글을 확인한다.
3. 게시판 페이지로 이동해 게시글 목록을 확인한다.
4. 글 작성이 필요하면 회원가입 또는 로그인을 진행한다.

### 회원 흐름

1. 사용자가 회원가입 페이지에서 계정을 만든다.
2. 로그인 페이지에서 이메일과 비밀번호로 로그인한다.
3. 게시판에서 글을 작성한다.
4. 프로필 페이지에서 내 정보와 내가 작성한 글을 확인한다.
5. 필요하면 본인 게시글을 수정하거나 삭제한다.
6. 게시글 상세 페이지에서 댓글 또는 대댓글을 작성한다.
7. 댓글 또는 대댓글에 좋아요, 싫어요 반응을 남긴다.

### 관리자 흐름

1. 관리자가 관리자 로그인 페이지에서 어드민 계정으로 로그인한다.
2. 서버가 계정의 `role` 값이 `admin`인지 확인한다.
3. 관리자 권한이 확인되면 관리자 페이지로 접근한다.
4. 관리자가 유저가 생성한 게시물 목록을 조회한다.
5. 운영 정책에 맞지 않는 게시물을 수정 또는 삭제한다.
6. 관리자가 댓글과 대댓글 목록을 조회한다.
7. 부적절한 댓글 또는 대댓글을 삭제한다.

## 15. 인증 및 보안 정책

- 비밀번호는 평문 저장을 금지하고 bcrypt로 해싱한다.
- 로그인 성공 시 JWT Access Token을 발급한다.
- 보호된 API는 Authorization 헤더의 Bearer Token을 검증한다.
- 관리자 API는 JWT 검증 후 `role` 값이 `admin`인지 추가 확인한다.
- 관리자 로그인은 어드민 계정만 성공 처리한다.
- 일반 회원 계정은 관리자 로그인 및 관리자 페이지 접근이 불가능해야 한다.
- 프론트엔드의 관리자 라우트 차단은 편의 기능이며, 최종 권한 검증은 백엔드에서 반드시 수행한다.
- 게시글 수정 및 삭제는 작성자 본인만 가능하다.
- 단, 관리자 API를 통한 게시물 수정 및 삭제는 관리자 권한으로 허용한다.
- 댓글과 대댓글 작성은 로그인 사용자만 가능하다.
- 댓글과 대댓글 수정 및 삭제는 작성자 본인만 가능하다.
- 단, 관리자 API를 통한 댓글 및 대댓글 삭제는 관리자 권한으로 허용한다.
- 댓글과 대댓글 좋아요/싫어요는 로그인 사용자만 가능하다.
- 하나의 사용자는 하나의 댓글 또는 대댓글에 하나의 반응만 남길 수 있다.
- 회원가입 시 이메일 중복을 검사한다.
- API 요청값은 서버에서 필수 검증한다.
- 운영 환경에서는 `.env` 파일로 JWT_SECRET, DB 접속 정보를 관리한다.

## 17. 성공 지표

- 사용자가 회원가입과 로그인을 정상적으로 완료할 수 있다.
- 로그인한 사용자가 게시글을 작성, 수정, 삭제할 수 있다.
- 사용자는 본인이 작성한 게시글만 수정, 삭제할 수 있다.
- 게시글 상세 페이지에서 조회수가 정상적으로 증가한다.
- 로그인한 사용자는 댓글과 대댓글을 작성할 수 있다.
- 로그인한 사용자는 댓글과 대댓글에 좋아요 또는 싫어요를 남길 수 있다.
- 비회원은 제한된 기능만 사용할 수 있다.
- 관리자 페이지는 어드민 계정으로만 접근할 수 있다.
- 관리자는 유저가 생성한 게시물을 관리할 수 있다.
- 관리자는 유저가 생성한 댓글과 대댓글을 관리할 수 있다.
- 홈페이지에 필요한 이미지와 로고가 프로젝트 내부 자산으로 적용되어 있다.
- 모바일 화면에서 주요 페이지와 관리자 페이지를 사용할 수 있다.
- 버튼, 댓글, 대댓글, 좋아요/싫어요, 관리자 기능에 명확한 인터랙션 피드백이 있다.
- 프론트엔드와 백엔드 API가 명확히 분리되어 유지보수하기 쉽다.
